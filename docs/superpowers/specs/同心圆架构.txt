Prompt · Context · Harness 工程全景图（同心圆模型）顶端副标题：三层协同 · 智能体循环 · 持续迭代 · 可靠落地核心底层逻辑（左下角与底部总结）三者如何协同（同心圆模型说明）：Prompt 决定模型 “做什么”（任务定义与输出表达）Context 决定模型 “知道什么”（信息供给与知识基础）Harness 决定模型 “如何可靠地做，并持续变好”（运行保障与优化闭环）全景图一句话总结（最底部大字）：Prompt 决定模型“做什么”，Context 决定模型“知道什么”，Harness 决定模型“如何可靠地做，并持续变好”。一、 内层核心：Prompt Engineering（内层核心指令层）定位说明：设计与表达：定义模型的任务、行为与输出方式。单次调用的指令，包含角色、示例、思维链、格式约束。1. 核心目标（带打勾符号）✓ 清晰表达目标与约束✓ 引导模型思考与推理✓ 控制输出的格式与风格✓ 降低歧义与幻觉✓ 提升可控性与可复现性2. 设计原则 (6P)（带橙色小图标）🎯 Purpose：明确目标👤 Persona：设定角色📝 Process：分步撰写指令⚖️ Policy：规则/约束📄 Presentation：输出格式🔍 Proof：自检/验证3. 结构化 Prompt 框架（示例）# 角色 (Persona)：你是一位 {role}，具备 {background} 能力。# 目标 (Purpose)：你的任务是 {task}，要达成 {goal}。# 背景 (Context Hint)：相关背景信息：{context_hint}。# 步骤 (Process)：1. {step1}；2. {step2}。# 约束 (Policy)：- {constraint1}；- {constraint2}。# 输出格式 (Presentation)：请按以下 JSON 格式输出：```json {format_example} ```# 自检 (Proof)：在输出前，检查：{checks}。4. 常用技巧👥 少样本/多样本 (Few-shot)：提供示例引导输出风格。🧠 思维链 (CoT)：分步撰写推理，提升准确性。🕸️ 结构化输出 (Json/Yaml)：便于解析与后处理。🔄 自我反思 (Reflexion)：先想后答，检查并改进。🎭 角色扮演 (Role Play)：设定专业视角与语气。5. 反例：糟糕的 Prompt（带红色小叉 ❌）❌ 指令模糊：帮我写点东西。❌ 缺少约束：不限格式与边界。❌ 信息缺失：没给必要背景与条件。❌ 单步复杂：一步要求太多事情。二、 中层架构：Context Engineering（中层信息供给）定位说明：为模型提供相关、精炼、结构化的信息。1. 核心目标（带绿/蓝色打勾符号）✓ 提供相关信息，减少噪声✓ 优化信息组织与顺序✓ 突破上下文长度限制✓ 支持检索与长期记忆2. 信息来源（带图标与文字）📝 用户输入 / 对话历史📑 知识库 / 文档 / 网页🗄️ 结构化数据 / API🤖 工具执行结果💾 记忆（短期 / 长期）3. 上下文处理流程（横向动态箭头图）检索 (Retrieve)（从知识库获相关信息） $\rightarrow$ 筛选 (Select)（过滤噪声/去重） $\rightarrow$ 压缩 (Compress)（摘要/提炼/裁剪） $\rightarrow$ 注入 (Inject)（按相关性与重要性排序） $\rightarrow$ 排序 (Order) $\rightarrow$ 算头 (Inject)（放入提示词/上下文）4. 检索增强流程 (RAG 示例流程图)用户问题 (Question) $\rightarrow$向量检索 (Vector Search) $\rightarrow$Top-K 文档片段 (相关内容)，图示分支为：📄 Doc 1 ……📄 Doc 2 ……📄 Doc K ……$\rightarrow$ 重排 / 压缩 (Rerank/Compress) $\rightarrow$注入上下文 (Inject)5. 优化策略分块 (Chunk)：语义分块，避免信息分割。元数据过滤：按时间/来源/权限筛选。摘要与压缩：保留核心信息，减少 token。多路检索：混合召回 + 精排重排。记忆整理：滚动窗口 + 关键记忆。动态注入：按需注入，避免冗余。6. 反例：糟糕的 Context（带红色小叉 ❌）❌ 无关信息过多 $\rightarrow$ 稀释注意力。❌ 顺序混乱 $\rightarrow$ 重要信息被淹没。❌ 超长未压缩 $\rightarrow$ 超出上下文限制。❌ 过时信息 $\rightarrow$ 造成错误决策。三、 外层系统保障：Harness Engineering定位说明：让代理（Agent）可靠运行、持续改进与规模化。1. 环绕中层蓝圈的 13 个系统控制节点（从左侧顺时针排列）🔄 循环与控制 (Agent Guardrails)🔑 权限与隔离 (Permission)💰 成本与配额 (Cost & Quota)🚀 部署与发布 (Deploy & Release)📈 可扩展性 (Scalability)2. 右侧看盘板块 ①：Harness 核心能力图表🔄 循环控制 (Agent Loop)🛠️ 工具系统 (Tools)🛡️ 校验机制 (Guardrails)🎯 评估系统 (Eval)📺 日志与追踪 (Observability)⚙️ 配置与版本 (Config & Version)3. 右侧看盘板块 ②：Agent Loop 详细流程横向主流程：① 收集上下文（输入/检索/记忆） $\rightarrow$ ② 采取行动（生成/调用工具） $\rightarrow$ ③ 校验结果（评估/反思/修正）回路指向：在步骤 ③ 产生 反馈 (Feedback) 并指向左侧，循环提示 $\circlearrowleft$ Repeat Until Done（直到完成或达到停止条件）。4. 右侧看盘板块 ③：工具编排（示例图标与中英文）🔍 搜索 Search🧮 计算器 Calculator🏽 代码执行 Code🗄️ 数据库 Database🔗 API 调用 API📄 文件读写 File I/O💬 ［省略号图标，代表更多工具］5. 右侧看盘板块 ④：校验与评估 (Quality Gate 细分指标)校验 (Guardrails)内容安全幻觉检测格式校验敏感信息过滤评估 (Eval)准确性（正确率）相关性（召回/命中）有用性（用户反馈）自动化测试集观察 (Observability)日志 / Trace指标 / 告警成本 / 延迟监控可视化 / 回放6. 右侧看盘板块 ⑤：反例：糟糕的 Harness（带红色小叉 ❌）❌ 无死循环：无停止条件与超时控制。（注：此处原文“无死循环”实为图表笔误，结合语义应指“产生死循环/无循环控制”）❌ 无校验：错误输出直接交付。❌ 不可观测：出错难定位与复盘。❌ 无评估：无法度量与持续改进。四、 底部工程落地方案1. 典型应用流程（端到端横向链路）用户输入 $\rightarrow$ 意图理解 (Prompt) $\rightarrow$ 检索上下文 (Context)
